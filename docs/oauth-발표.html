The Problem

왜 OAuth가 필요한가?

인화 앱이 내 구글 사진을 가져오려 한다.
가장 단순한(그리고 끔찍한) 방법:

"인화 앱에 구글 아이디·비밀번호를 그냥 입력하세요."
  • 앱이 내 비밀번호를 그대로 알게 됨 → 이메일·설정까지 뭐든 가능
  • 권한을 "사진 읽기만" 으로 제한 불가 (전부 아니면 전무)
  • 앱만 끊으려면 비밀번호를 바꿔야 함 → 다른 앱도 다 끊김
OAuth = 비밀번호를 안 넘기고, 딱 필요한 권한만 위임하는 방법
Mental Model

핵심 비유: 호텔 카드키 🔑

  • 체크인 때 신분증(비밀번호)은 프런트에만 보여줌
  • 프런트가 카드키(액세스 토큰)를 발급
  • 카드키는 내 방 문만 열 수 있고 (scope)
  • 체크아웃하면 만료되고 (수명)
  • 잃어버려도 신분증은 안전, 카드키만 재발급
앱에게 넘기는 건 신분증이 아니라 카드키.

이 한 문장이 OAuth의 전부.
나머지는 이걸 안전하게 구현하는 장치.
The Cast

등장인물 4명 + scope

🙋
Resource Owner데이터 주인 = 사용자 (투숙객)
📱
Client데이터를 원하는 앱 = 인화앱
🏨
Authorization Server로그인·동의·토큰 발급 (프런트 데스크)
🚪
Resource Server토큰 확인 후 데이터 제공 (객실 문)
scope = 허용할 작업의 범위. "사진 읽기" 처럼 권한을 좁게 자른다. 발급소(Auth Server)와 검문소(Resource Server)는 같은 회사(구글)라도 역할이 다르다.
Auth vs Authz

인증(Authentication) vs 인가(Authorization)

Authentication

"너 누구야?"

신원 확인 = 로그인

Authorization

"너 뭘 해도 돼?"

권한 부여 = 위임

OAuth 2.x 는 원래 '인가' 프로토콜. 앞의 예시는 전부 "사진 읽을 권한 위임" 이야기였다.
"구글로 로그인"(인증)을 하려면 → OAuth 위에 OIDC 를 얹는다.
Access Token = API용(뭘 할 수 있나) · ID Token = 로그인용(누구인가)
The Flow

Authorization Code Flow

0
사용자 → 앱
"구글로 로그인" 클릭 → /oauth/google/login
1
앱 → 구글
state 생성 → 세션 저장 → 구글 로그인 화면으로 302 리다이렉트
2
사용자 ↔ 구글
구글에서 로그인 + "사진 읽기 허용?" 동의
3
구글 → 앱
콜백으로 되돌려보냄 → ?code=…&state=…
4
앱 (내부)
state 대조 (통과해야 다음으로)
5
앱 ↔ 구글
code + client_secret → access_token 교환
6
앱 ↔ 구글 API
토큰으로 사진 데이터 요청
앞길 — 브라우저 경유(노출 위험): state·code만 흐름 뒷길 — 서버↔서버(안전): secret·token이 오감
Why an intermediate code?

왜 토큰을 바로 안 주고 code 를 먼저 줄까?

  • 콜백(3)은 브라우저 URL을 거친다 → 히스토리·로그에 남을 위험
  • 그래서 그 길엔 진짜 토큰 대신 일회용 code만 흘려보냄
  • 진짜 토큰 교환은 서버가 뒷길에서 client_secret과 함께 조용히
"위험한 앞길로는 임시 code만,
안전한 뒷길로는 진짜 token만."
client_id = 가게 이름(공개)
client_secret = 금고 비밀번호(서버만)
Defense #1 · CSRF

state — "이 콜백, 내가 시작한 게 맞아?"

막는 공격: 로그인 CSRF (남의 계정을 나에게 떠넘기기)

!
공격자가 자기 계정으로 로그인 시작 → 콜백 URL을 손에 쥠
!
그 URL을 피해자에게 던져 클릭 유도
!
피해자가 공격자 계정에 로그인된 채 개인정보를 올림 😱
방어: 시작할 때 랜덤 state를 만들어
① URL에 실어 보내고 ② 세션에도 저장.

콜백에서 URL의 state == 세션의 state 여야 통과.
공격자 URL엔 그 브라우저 세션의 state가 없음 → 불일치 → 거절.
= "URL만 훔쳐선 아무것도 못 한다."
Defense #1 · Code

state — 실제 코드

OAuthController.login() — 시작 시 심는다
// ① 랜덤 state 생성
String state = generateState();

// ② 서버 세션에 저장 (브라우저엔 세션ID만)
session.setAttribute(KEY + provider, state);

// ③ state 를 URL에 실어 리다이렉트
.queryParam("state", state)
OAuthController.callback() — 돌아올 때 대조
String saved =
  (String) session.getAttribute(KEY+provider);
session.removeAttribute(KEY+provider); // 1회용

if (saved == null || !saved.equals(state)) {
    return "거절: state 불일치";
}
// 통과 → 이제 code 를 토큰으로 교환
검증 주체 = 우리 서버
Defense #2 · Motivation

그런데 code 자체가 새면?

  • 백엔드 앱: code를 훔쳐도 client_secret 없이는 교환 불가 → 안전
  • 하지만 secret을 숨길 수 없는 앱이 있다
  • React SPA — 소스가 브라우저에 다 내려감 → devtools로 secret 노출
  • 모바일 앱 — 파일 뜯으면 secret 나옴
이런 Public Client는 secret이 방패가 못 된다 → code만 훔치면 계정 탈취
PKCE (픽시)

= client_secret 없이도
"code를 훔쳐가도 못 쓰게" 만드는 방법.

비결: 로그인마다 만드는 일회용 비밀번호로 code를 잠근다.
Defense #2 · Idea

PKCE — 열쇠와 자물쇠

🔑

code_verifier

랜덤 비밀값

열쇠. 내 손(세션)에만 쥔다.
네트워크로 안 나감.

🔒

code_challenge

SHA256(code_verifier)

자물쇠. 로그인 시작 때 이것만 보냄.
노출돼도 안전.

해시는 단방향 — 자물쇠(🔒)를 훔쳐봐도 열쇠(🔑)를 되돌릴 수 없다.
Defense #2 · Flow

PKCE 흐름 — 자물쇠 먼저, 열쇠 나중

1
앱 → 구글
열쇠 생성→세션 저장. URL엔 code_challenge(자물쇠) + method=S256 만
2
구글
code 발급 시 자물쇠를 함께 기억해둠
3
구글 → 앱
콜백으로 code 돌려줌 (여기까진 기존과 동일)
4
앱 ↔ 구글
토큰 교환 요청에 code_verifier(열쇠 원본) 제시
5
구글 (검증)
SHA256(열쇠) == 저장된 자물쇠 ? → 맞아야 토큰 발급
공격자: code ✅ 훔침 / 열쇠 ❌ 없음 → 자물쇠에서 열쇠 못 만듦 → 토큰 교환 거부
Defense #2 · Code

PKCE — 실제 코드 (바뀐 곳 3군데)

login() — 열쇠 만들고 자물쇠만 보냄
String v = generateCodeVerifier();   // 🔑
String ch = createCodeChallenge(v); // 🔒
session.setAttribute(VKEY+p, v); // 열쇠는 세션에

.queryParam("code_challenge", ch)     // 자물쇠만
.queryParam("code_challenge_method","S256")
callback() — 세션에서 열쇠 꺼내 전달
String v =(String)session.getAttribute(VKEY+p);
exchangeCodeForToken(config, code, v);
OAuthClient.exchangeCodeForToken()
form.add("grant_type","authorization_code");
form.add("code", code);
form.add("client_secret", secret);
form.add("code_verifier", v); // 🔑 열쇠 제시
createCodeChallenge =
BASE64URL( SHA-256( verifier ) )
→ 단방향이라 URL에 노출돼도 안전
검증 주체 = 제공자(구글)
Summary

state vs PKCE

state 🛡️PKCE 🔐
막는 공격CSRF (남이 떠넘긴 로그인)code 탈취/가로채기
묶는 대상콜백 ↔ 내 브라우저 세션code ↔ 원본 verifier
방식같은 값 돌려받아 대조해시 먼저, 원본 나중 (단방향)
검증 주체우리 서버 (콜백에서)제공자 (토큰 교환에서)
노출돼도 OKstate (세션이 방패)challenge (verifier가 열쇠)
막는 공격이 달라서 함께 쓴다. OAuth 2.1 부터는 모든 클라이언트에 PKCE 권장.
Wrap up

한 장 정리

위임

OAuth

비밀번호가 아니라 카드키(토큰)를 넘긴다. code→token→API.

인증

OIDC

"누구인가"가 필요하면 OAuth 위에 ID Token을 얹는다.

방어

state + PKCE

state=요청 위조 방어, PKCE=code 탈취 방어. 항상 함께.
