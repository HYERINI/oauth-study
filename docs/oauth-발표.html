Why & Mental Model

왜 OAuth인가 + 호텔 카드키 비유

가장 단순한(끔찍한) 방법: "인화앱에 구글 비밀번호를 그냥 입력하세요"
  • 앱이 비밀번호를 그대로 알게 됨 → 이메일·설정까지 뭐든
  • 권한을 "사진 읽기만"으로 제한 불가 (전부 아니면 전무)
  • 앱만 끊으려면 비밀번호를 바꿔야 → 다른 앱도 끊김
  • 체크인 때 신분증(비밀번호)은 프런트에만 보여줌
  • 프런트가 카드키(토큰)를 발급
  • 카드키는 내 방만 열고(scope) · 만료되고(수명) · 재발급 가능
앱에게 넘기는 건 신분증이 아니라 카드키 🔑
= 비밀번호 없이 필요한 권한만 위임 — 이게 OAuth의 전부.

The Flow · Sequence

Authorization Code Flow — 왜 code 먼저? 앞길엔 code만, 뒷길에서 token 교환

앞길 — 브라우저 경유(노출 위험): state·code만 뒷길 — 서버↔서버(안전): secret·token ─ ─ 는 응답(되돌아오는 방향)
Defense #1 · state (CSRF)

state — "이 콜백, 내가 시작한 게 맞아?"

CSRF 공격: 공격자가 자기 계정으로 로그인 시작 → 콜백 URL을 피해자에게 던져 클릭 유도 → 피해자가 공격자 계정에 로그인된 채 개인정보를 올림 😱
방어: 시작 때 랜덤 state를 ① URL에 싣고 ② 세션에도 저장. 콜백에서 URL의 state == 세션의 state여야 통과. 공격자 URL엔 그 세션의 state가 없어 거절.
검증 주체 = 우리 서버 · "URL만 훔쳐선 아무것도 못 한다"
OAuthController.login() — 시작 시 심는다
// ① 랜덤 state 생성
String state = generateState();
// ② 서버 세션에 저장(브라우저엔 세션ID만)
session.setAttribute(KEY + provider, state);
// ③ state 를 URL에 실어 리다이렉트
.queryParam("state", state)
callback() — 돌아올 때 대조
String saved =(String)session.getAttribute(KEY+p);
session.removeAttribute(KEY+p); // 1회용
if(saved==null || !saved.equals(state))
    return "거절: state 불일치";
// 통과 → 이제 code 를 토큰으로 교환
Defense #2 · PKCE

PKCE — 왜 필요? + 열쇠 🔑 / 자물쇠 🔒

  • 백엔드: code 훔쳐도 client_secret 없으면 교환 불가 → 안전
  • secret 못 숨기는 앱: React SPA·모바일 → devtools·파일로 노출
  • 이런 Public Client는 code만 훔치면 계정 탈취
PKCE = secret 없이도 "code 훔쳐가도 못 쓰게".
로그인마다 만드는 일회용 비밀번호로 code를 잠근다.
🔑

code_verifier

랜덤 비밀값

열쇠. 세션에만 쥔다.
네트워크로 안 나감.

🔒

code_challenge

SHA256(verifier)

자물쇠. 로그인 때
이것만 보냄.

해시는 단방향 — 자물쇠(🔒)에서 열쇠(🔑)를 못 되돌린다.
Defense #2 · Sequence

PKCE 흐름 — 자물쇠 먼저 🔒, 열쇠 나중 🔑

공격자: code ✅ 훔침 / 열쇠 🔑 ❌ 없음 → 자물쇠(🔒)에서 열쇠 못 만듦(단방향) → 토큰 교환 거부
Defense #2 · Code

PKCE — 실제 코드 (바뀐 곳 3군데)

login() — 열쇠 만들고 자물쇠만 보냄
String v = generateCodeVerifier();   // 🔑
String ch = createCodeChallenge(v); // 🔒
session.setAttribute(VKEY+p, v); // 열쇠는 세션에
.queryParam("code_challenge", ch)     // 자물쇠만
.queryParam("code_challenge_method","S256")
callback() — 세션에서 열쇠 꺼내 전달
String v =(String)session.getAttribute(VKEY+p);
exchangeCodeForToken(config, code, v);
OAuthClient.exchangeCodeForToken()
form.add("grant_type","authorization_code");
form.add("code", code);
form.add("client_secret", secret);
form.add("code_verifier", v); // 🔑 열쇠 제시
createCodeChallenge = BASE64URL( SHA-256( verifier ) )
→ 단방향이라 URL에 노출돼도 안전
검증 주체 = 제공자(구글) · state와 검증 주체가 다름
Summary

state vs PKCE · 한 장 정리

state 🛡️PKCE 🔐
막는 공격CSRF (남이 떠넘긴 로그인)code 탈취/가로채기
묶는 대상콜백 ↔ 내 브라우저 세션code ↔ 원본 verifier
방식같은 값 돌려받아 대조해시 먼저, 원본 나중 (단방향)
검증 주체우리 서버 (콜백)제공자 (토큰 교환)
위임

OAuth

비밀번호가 아니라 카드키(토큰)를 넘긴다. code→token→API.

인증

OIDC

"누구인가"가 필요하면 OAuth 위에 ID Token을 얹는다.

방어

state + PKCE

state=요청 위조, PKCE=code 탈취. OAuth 2.1은 항상 함께.
